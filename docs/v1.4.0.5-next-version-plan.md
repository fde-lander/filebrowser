## Chapter 2: Backend Background Compress System (CRITICAL)

### 2.1 Problem

**CRITICAL: After first SSE disconnect, ALL subsequent compress attempts fail permanently.**

Current compress runs in a goroutine BUT:
- SSE connection is tied to the frontend dialog
- When user closes dialog: SSE disconnects, progress lost
- When network drops: SSE disconnects, error shown
- **Once SSE disconnects with "Connection lost" error, the entire compress system becomes broken** - all new compress attempts fail within 1-2 seconds with the same error
- Backend goroutine may continue but progressMgr/taskID state gets corrupted
- This means the OLD SSE-based system is fundamentally broken and must be REPLACED entirely

### 2.1b Root Cause Investigation (for next AI)

The permanent failure after first SSE disconnect suggests one of:
1. **progressMgr channel deadlock**: SSE handler blocks on channel read, goroutine can't send progress, deadlock cascades
2. **TaskID collision**: progressMgr.createTask() uses timestamp, but closeTask may not be called properly after disconnect
3. **Goroutine resource leak**: Previous job goroutine still running, holding progressMgr resources
4. **SSE handler blocks forever**: After client disconnect, SSE handler goroutine never returns, consuming a server thread

Next AI MUST investigate by:
- Reading progressMgr implementation (createTask, closeTask, getProgressChan, getDoneChan)
- Check if channels are properly closed on client disconnect
- Check if progressMgr has cleanup on task completion
- Check if there's a goroutine leak detector

### 2.2 Requirements

**MASTER DIRECTIVE: Rebuild compress system entirely. Replace SSE with polling.**

1. Compress MUST run entirely backend-side as a persistent background task
2. **ABANDON SSE entirely** - frontend polls status via API every 3 seconds
3. Frontend is ONLY a status viewer - it polls, displays, can disconnect freely
4. When user closes dialog: backend continues compressing
5. When user re-opens "Compress Images": if a job is running, show its progress directly
6. Only ONE compress job at a time (new requests rejected with message)
7. Backup must complete BEFORE any compression starts (existing design, keep)
8. Every file MUST be processed through the pipeline (backup -> compress -> replace)
9. Progress state must be queryable via API polling (NOT SSE)

### 2.3 Architecture Design

**Backend changes (compress.go):**

- Add a global `compressJobManager` singleton (package-level struct)
  - Holds current running job: taskID, totalFiles, processed, skipped, failed,
    backupPath, backupFallback, status (running/done/failed), startTime
  - Mutex-protected for concurrent access
  - Methods: `startJob()`, `getJobStatus()`, `isRunning()`, `getProgress()`
  - Survives HTTP request lifecycle - state persists until next job starts

- compressHandler: Check if job already running
  ```go
  if compressJobManager.isRunning() {
      return 409 Conflict, "A compress job is already running. Wait for it to finish."
  }
  ```
  If not running: create job, start goroutine, return taskId immediately

- compressProgressHandler: SSE stream reads from compressJobManager
  - Client can connect/disconnect freely
  - Each connection gets current snapshot + live updates
  - No dependency on request lifecycle

- NEW endpoint: GET /compress-images/status
  - Returns JSON: { running, taskId, total, processed, skipped, failed, backupPath, status }
  - Allows frontend to check if a job is running WITHOUT SSE
  - Used when opening compress dialog to detect running jobs

- When goroutine finishes: update compressJobManager status to "done"
  - Keep result data for frontend to query
  - Clear when next job starts

**Frontend changes (CompressImages.vue + compress.js):**

- **REMOVE SSE subscription entirely** (subscribeProgress function + EventSource)
- **REPLACE with polling**: setInterval every 3 seconds, call GET /compress-images/status

```js
// Replace subscribeProgress with pollStatus
function pollStatus(taskId, handlers) {
  const poll = async () => {
    try {
      const status = await getCompressStatus();
      if (status.running && status.processed < status.total) {
        handlers.onProgress({
          processed: status.processed,
          total: status.total,
          current: status.currentFile,
          percent: status.percent,
          skipped: status.skipped,
          failed: status.failed,
        });
      } else if (!status.running) {
        clearInterval(intervalId);
        handlers.onComplete({
          success: status.success,
          skipped: status.skipped,
          failed: status.failed,
          backupPath: status.backupPath,
          backupFallback: status.backupFallback,
        });
      }
    } catch (err) {
      // Network error - keep polling, don't abort
      console.error("Status poll error:", err);
    }
  };
  const intervalId = setInterval(poll, 3000);
  return { close: () => clearInterval(intervalId) };
}
```

- On dialog open: call GET /compress-images/status
  - If running: switch to progress view immediately, start polling
  - If not running: normal file selection view

- Progress view: user can close dialog
  - beforeUnmount: clearInterval (stop polling), backend continues regardless
  - No SSE to disconnect - just stop the timer

- Re-open dialog: status check -> if still running -> resume progress view
  - If job completed while away: show completion notification

- Add "Cancel Job" button in progress view (optional, MASTER can decide)
  - Would need backend endpoint: POST /compress-images/cancel

### 2.4 Polling Reliability (Replaces SSE)

- Frontend polls GET /compress-images/status every 3 seconds
- If network drops: poll fails silently, keeps retrying (no error to user)
- When network returns: poll resumes, shows current backend state
- No heartbeat needed (HTTP request/response is stateless)
- No permanent failure possible (each poll is independent)
- Backend has NO dependency on frontend connection
- compressJobManager holds all state, survives frontend disconnects

### 2.5 Single Instance Enforcement

- Backend: compressJobManager.isRunning() check in compressHandler
- Frontend: disable "Compress" button if status shows running
- Error message: "A compress job is already running ({processed}/{total}). Please wait."

## Chapter 5: Preview Original Image Quality Fix

### 5.1 Problem

Preview original image (left side) is very blurry - looks like a low-quality thumbnail.
Even in fullscreen mode, the original stays blurry.

### 5.2 Root Cause

In CompressImages.vue openPreview():
```js
this.previewUrls.original = getPreviewURL(file.source, file.path, file.modified);
```

getPreviewURL (resources.js line 843) builds URL without `size` parameter:
```
/api/resources/preview?path=...&key=...&source=...&inline=true
```

Without `size=original`, the backend returns a generated thumbnail (small, blurry).

Compare with Preview.vue raw() computed (line 192-197):
```js
return `${resourcesApi.getPreviewURL(source, path, modified)}&size=original`;
```

Preview.vue adds `&size=original` to get full resolution.

### 5.3 Fix

In CompressImages.vue openPreview(), append `&size=original` to the URL:

```js
// Get original image URL (full resolution, not thumbnail)
const previewUrl = getPreviewURL(file.source, file.path, file.modified);
this.previewUrls.original = previewUrl + '&size=original';
```

This ensures the left side of preview shows the SAME quality as the actual
image viewer. Fullscreen overlay will also show full resolution.

### 5.4 Alternative: Use getDownloadURL

If size=original still has quality issues, use download URL instead:
```js
import { getDownloadURL } from "@/api/resources.js";
this.previewUrls.original = getDownloadURL(file.source, file.path, true);
```

getDownloadURL returns a direct file download URL (inline=true for browser display).
This bypasses the preview thumbnail system entirely and serves the raw file.

### 5.5 Compressed Preview Quality

Compressed preview (right side) uses blob URL from backend compressPreviewHandler.
The backend compresses the full image and returns the compressed binary.
This side is already full resolution (compressed), confirmed working by MASTER.

## Chapter 6: Progress Accuracy + Polling Replace SSE

### 6.1 Problem

Compressing 248 images, progress stuck at 149/248, then SSE disconnected.
After this error, ALL subsequent compress attempts fail within 1-2 seconds.
The entire SSE-based progress system is fundamentally broken and must be replaced.

### 6.2 Root Cause Analysis

1. SSE handler goroutine may block forever after client disconnect
2. progressMgr channels may deadlock after disconnect
3. Once progressMgr is corrupted, new jobs inherit bad state
4. SSE is the wrong architecture for long-running background tasks

### 6.3 Fix: Polling Architecture (see Chapter 2)

**SSE is REMOVED entirely.** Replaced with 3-second polling via GET /compress-images/status.

**Backend progress accuracy (compress.go):**

Move progress update to AFTER each file is processed:
```go
for _, filePath := range realPaths {
    result := compressImage(filePath, req.Level, req.Quality)
    if result.Err != nil { failed++; continue }
    if result.Skipped { skipped++; continue }
    // Write + replace...
    success++
    // Update job manager AFTER processing
    compressJobManager.updateProgress(success + skipped + failed, skipped, failed, filepath.Base(filePath))
}
```

processed count = success + skipped + failed (accurate, not loop index)

### 6.4 No SSE = No Heartbeat Needed

Polling is stateless HTTP. No connection to maintain. No heartbeat required.
Network drops just mean missed polls - backend continues regardless.

### 6.5 Frontend Reconnection

No reconnection needed. Polling resumes automatically when network returns.
Each poll is an independent HTTP GET - no state to recover.

### 8.10 Master Design Refinement - Queue-Based Compress System (NEW - 2026-07-13)

**MASTER design clarification** (correction to Ch2 earlier proposal):

Previous Ch2 said "Only ONE compress job at a time" (single instance).
MASTER now refines this to a **QUEUE-based system**:

**Design**:
1. Backend maintains a compress QUEUE (not single job)
2. Queue processes ONE FILE at a time (serial within queue)
3. User can ALWAYS add more files to the queue while one is running
4. If compress is running, opening "Compress Images" dialog shows:
   - On the RIGHT side of dialog: current queue status (current file, progress X/Y)
   - Left side: normal file selection (can add more files)
   - Click "Compress" while running -> API call appends files to queue
5. Multiple compress jobs can be queued, processed one-by-one
6. User can close dialog anytime - queue continues in background

**Why this is better than "single instance"**:
- More humane UX - users don't get blocked by "job already running" error
- Can batch multiple folders at once without waiting
- Can check status while compress is running
- Adding files to queue never fails

**API changes**:
- POST /compress-images (existing): append files to queue
  - If queue empty: start processing immediately
  - If queue has items: append to end, return position in queue
- GET /compress-images/queue (new): get full queue status
  - Returns: { running, currentFile, processed, total, queue: [{id, files, status}] }
- POST /compress-images/queue (new): alternative endpoint for adding to queue

**Backend architecture (refined)**:
- compressQueueManager (instead of compressJobManager)
  - Queue: []QueueItem (each item is a compress request with files + settings)
  - Worker goroutine: processes queue one item at a time
  - Status: currentItem, processed, total, queueLength
- Single worker goroutine (single core, no parallelism anyway)
- Worker loops through queue items, for each item compresses all files

**Frontend changes**:
- Dialog layout (refined per MASTER feedback):
  - Top: normal file selection area (full width, names must be readable)
  - Bottom: thin queue status bar (NOT a side panel that takes up horizontal space)
  - OR: collapsible bottom panel (expanded only when queue has items)
- Queue status bar shows (compact, 1-2 lines):
  - "Currently processing: X.jpg (149/250)"
  - "Queued: folder2 (50 images), folder3 (30 images)"
  - Total queue summary: "Queue: 280 images in 3 batches"
- Click "Compress" button always works - adds to queue if running
- No "already running" error message needed

   - Cannot speed up by adding workers
   - Background processing is the solution (user can do other things)

2. **Memory is NOT a constraint** - 2GB server has plenty of headroom
   - Original concern about ZRAM/SWAP thrashing was unfounded
   - Can afford to load large images into memory
   - No need for memory pooling or streaming

3. **Queue system (Ch8.10) is the right approach** - single worker processing
   serially is optimal for this hardware
   - Multiple parallel workers would only add context switching overhead

4. **Processing speed is ACCEPTABLE** - now that backend runs in background:
   - User can close dialog and do other things
   - Compression time is no longer user-blocking
   - Quality preservation is MORE important than speed optimization

5. **Background processing is essential** - while user waits:
   - CPU is fully used (can't do other CPU-heavy work simultaneously)
   - But light tasks (browsing files, viewing images already compressed) still work
   - Memory headroom allows other operations to coexist

### 8.11 Implementation Considerations for Next AI

1. Queue data structure: slice of QueueItem structs (mutex-protected)
2. Worker goroutine: select on queue channel, process one item at a time
3. QueueItem struct: { id, files, level, quality, source, status, addedTime }
