## Chapter 1: Transition Fix Rollback (CRITICAL)

### 1.1 Problem

The transition fix (commit 44616283) attempted to fix anonymous/share user
image navigation by modifying 6 points in ExtendedImage.vue. Result:

- **Logged-in users:** Visual quality regression. After transition completes,
  image shows a brief quality shift (like loading from low-res to high-res).
  This causes a subtle but noticeable flicker. V1.4.0.4 does NOT have this.
- **Anonymous/share users:** No confirmed improvement.

### 1.2 Root Cause Analysis (for v1.4.0.6)

The 6 fixes changed the transition engine's timing and error handling:
1. waitForDecode no longer swallows errors → decode timing changed
2. navigateToImage .catch() aborts transition → different flow
3. onImageError guard → changes error display behavior
4. finishTransition verifies image loaded → delays flag setting
5. Old buffer src cleared → may cause re-fetch (quality shift)
6. Default 'fade' + imgB @load → timing changes

The "brief quality shift" is likely caused by Fix 4 (finishTransition
verifies loaded) or Fix 5 (clearing src triggers re-fetch/re-decode).
The old buffer's src was kept for a reason — the browser retains the
decoded bitmap, allowing instant display. Clearing it forces re-fetch.

### 1.3 Decision

**ROLLBACK to v1.4.0.4-hotfix transition behavior.**
All 6 changes reverted. ExtendedImage.vue + Profile.vue restored exactly.

### 1.4 v1.4.0.6 Direction

Do NOT modify transition engine again unless a fundamentally different
approach is found. The current approach of patching error handling makes
things worse for logged-in users. Possible alternatives:
- Investigate if anonymous user issue is actually CORS/auth related (not
  transition logic)
- Consider server-side fix (different image serving for share URLs)
- Accept current v1.4.0.4 transition behavior as-is

## Chapter 2: Queue Compress System Bugs (CRITICAL)

### 2.1 Bug A: Folder-level Compress Does Nothing (BLOCKER)

**Symptom:** Right-click folder → Compress Images → dialog shows file list
(expanded correctly) → click Compress button → NOTHING happens.
No API call, no error, no progress. Backend has NO logs.

**Expected:** Should compress all images in the folder.

**Root Cause (preliminary):** The expandItems() method populates
expandedItems[] for display, but the doCompress() method likely still
reads from this.items (or this.selectedFiles mapped to original items)
instead of expandedItems. When folder items are passed to the API,
they're directory paths, not file paths. The backend may silently fail
or the frontend may not send the request at all.

**Fix Direction:** doCompress() must use expandedItems when available.
The API call must send actual file paths, not directory paths.

### 2.2 Bug B: Progress Status Window Lost After Close (CRITICAL)

**Symptom:** Compress starts → progress shows → close dialog → reopen
Compress Images → NO status shown. Backend IS still running.
User has NO way to know backend is compressing.

**Expected:** Reopening compress dialog should show current queue status
(running, progress X/Y, current file).

**Root Cause (preliminary):** The polling (setInterval) is created in
doCompress() and cleared when dialog closes (beforeUnmount clears timer).
When dialog reopens, it does NOT check if backend is already running.
No pollStatus() call on mount to detect existing queue.

**Fix Direction:** mounted() should call pollStatus() to check if backend
has an active job. If so, show progress UI immediately.

### 2.3 Bug C: Progress Stuck at N-1/N (CRITICAL)

**Symptom:** Compress running → shows e.g. "0/3" → progresses to "2/3"
→ stuck there forever. Backend has FINISHED but UI still shows 2/3.
The polling keeps requesting but never gets "completed" status.

**Expected:** When backend finishes, UI should show completion + notification.

**Root Cause (preliminary):** The compressWorker goroutine calls
finishCurrent() which sets status to "completed". But then
setWorkerActive(false) runs immediately after (in the defer),
which sets status to "idle" because queue is empty. This overwrites
"completed" before the frontend's 3-second poll can catch it.

**Fix Direction:** Need to preserve "completed" state long enough for
frontend to catch it. Options:
- Keep "completed" status for 30 seconds before resetting to "idle"
- OR: Add a separate "lastResult" field that doesn't get overwritten
- OR: Frontend treats "idle" after "running" as completion signal

### 2.4 Bug D: Queue Total Count Confusion

**Symptom:** Compress batch 1 running. Open dialog, select more files,
compress → added to queue. But UI shows batch 1's total count until
batch 1 finishes, then shows batch 2 count (e.g. 0/3, stuck at 2/3).

**Expected:** Should show overall queue progress or current batch clearly.

**Fix Direction:** Status API should clearly separate "current batch"
from "queue". Frontend should show both.

### 2.5 Bug E: No Success Notification

**Symptom:** Compression completes but no success notification shown.

**Root Cause:** Related to Bug C — if polling never catches "completed"
status, the notification code never runs.

### 2.6 Bug F: UI Changes Had No Visual Effect (CRITICAL - INVESTIGATED)

**Symptom:** MASTER reports compress dialog UI looks identical to v1.4.0.4.
Preview checkbox still in old position. No queue status bar.

**Investigation Result (2026-07-14 main agent code trace):**

1. **Preview checkbox NOT moved (CONFIRMED):**
   - Plan Task 6 Step 6.3 said to move checkbox from L133-139 to after L75
   - In actual code execution, this step was SKIPPED entirely
   - Checkbox remains at L126-132 (above Options, same as v1.4.0.4)
   - Root cause: AI execution error — step was in plan but not applied

2. **Queue status bar template NOT added:**
   - Plan mentioned "thin bottom bar showing queue status"
   - No template HTML for this was ever written
   - Only the backend API (GET /compress-images/status) exists

3. **CSS duplicate definitions found:**
   - .compress-preview-img defined TWICE: L713 (max-height: 150px, for
     file list thumbnails) and L865 (max-height: 100%, for preview overlay)
   - .compress-preview-size defined TWICE: L721 and L877
   - These duplicates may cause CSS cascade issues depending on specificity

**MASTER's Quote:** "压缩图片窗口的UI确认一下点解修改了完全无效，跟1.4.0.4一模一样"

**Fix Direction:**
1. Actually MOVE the checkbox HTML block (was skipped)

4. VERIFY changes by grepping for new positions after patch

## Chapter 4: MASTER Development Directives for v1.4.0.6

### 4.1 Queue List API (MASTER Request)

**MASTER Quote:** "后端压缩系统係几好，但系要有一个功能可以取得所有加入到队列的压缩列表咁样找回状态"

**Requirement:** A new API endpoint that returns ALL items in the queue
(not just current job status). This allows the frontend to show:
- Full queue list: each item with its files, status (queued/processing/done)
- Current job progress within the context of the full queue
- When dialog reopens, immediately fetch full queue state

**Proposed API:** GET /compress-images/queue
Returns: { queue: [{ id, files, level, status, progress }], current: {...} }

This is the key to solving Bug B (status lost after close) — the frontend
can ALWAYS fetch the full queue state on dialog mount, regardless of
whether a previous dialog started the job.

### 4.2 Folder Compress Must Work (MASTER Directive)

**MASTER Quote:** "文件夹压缩功能个度肯定係要修复好。多个文件夹也要。"

**Requirement:** Right-click one or multiple folders → Compress Images →
all images within those folders must be compressed. This is the core use
case. Must support:
- Single folder selection
- Multiple folder selection
- Mixed (files + folders)
- Nested subfolders (recursive expansion)

Bug A (folder compress does nothing) is P0 BLOCKER.

### 4.3 UI Investigation Recorded (MASTER Directive)

**MASTER Quote:** "压缩图片窗口的UI确认一下点解修改了完全无效"

**Finding:** Documented in Ch2.6 Bug F above. Three root causes:
1. Checkbox move step skipped during execution (AI error)
2. Queue status bar HTML never written
3. Duplicate CSS definitions may cause cascade conflicts

This is recorded for v1.4.0.6 fix with verification step after patching.
