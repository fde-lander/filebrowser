## Chapter 1: Mobile Tap Navigation Transition Quality Regression

### 1.1 Symptom

Mobile tap navigation now WORKS (Bug G fix successful - tap triggers navigation).
HOWEVER, the visual transition quality during tap navigation is DEGRADED compared to:
- The original (pre-v1.4.0.6) behavior
- Swipe navigation (which remains smooth/silky)

**MASTER Quote:** "移动端的点击翻页功能喺有啦。但喺我总喺视觉上感觉距的渐变喺无原来那种咁丝滑的，对比度喺更大"
"点击翻页不能讲话失败，但喺我好灵敏，我察觉到距的过渡都系有比不上原来不搞的"
"滑动翻页正常，效果就喺以前个种好丝滑的"

### 1.2 Analysis

- Swipe navigation: smooth transition (unchanged, good)
- Tap navigation: works but transition has higher contrast / less silky

**Possible causes to investigate:**
1. Tap navigation goes through handleImageClick -> $emit('navigate-previous/next') -> parent changes src -> src watcher -> navigateToImage
2. Swipe navigation goes through finishEdgeGesture -> $emit('navigate-previous/next') -> same path
3. Both paths converge to the same emit, BUT:
   - Tap fires via click event (synthetic, after touchend)
   - Swipe fires via touchend directly
   - Timing difference: tap has ~300ms delay after touchend (browser click synthesis)
   - This delay may cause the transition engine to receive the src change at a different point in the render cycle
4. The conditional preventDefault in touchMove/touchEnd may subtly affect browser rendering pipeline
5. The `getters.previewType() !== 'image'` guard in nextPrevious.vue prev()/next() was added for Bug H - this SKIPS setNavigationTransitioning(true) for images. While this prevents the loading overlay, it may also affect the transition engine's timing/coordination

**Key question:** Does the transition quality difference exist because:
- (A) The click event fires at a different timing than swipe?
- (B) The setNavigationTransitioning skip affects transition coordination?
- (C) Some other interaction between the touchMove/touchEnd changes?

### 1.3 Fix Direction (NEEDS INVESTIGATION)

- Compare tap vs swipe code paths to find where they diverge
- Check if setNavigationTransitioning skip is causing the quality difference
- Consider: should tap navigation use the same code path as swipe?
- DO NOT modify transition engine (navigateToImage/swapBuffers/finishTransition)
- MUST preserve swipe quality (currently good)

## Chapter 3: Compress Status Bar + Control Features COMPLETELY MISSING (CRITICAL)

### 3.1 Symptom

After deploying v1.4.0.6-fde:
- NO global status bar appears during compression
- NO control buttons (pause/skip/cancel) visible in compress dialog
- NO progress recovery when reopening compress dialog
- Essentially ALL new compress features from v1.4.0.6 are invisible

**MASTER Quote:** "对于压缩的提示乜嘢的新功能完全无卵发现有"
"完全无任何新增的状态栏，全局都无提示"

### 3.2 Root Cause Analysis

**CompressStatusBar.vue was NEVER TRIGGERED:**
- The component was created and registered in Prompts.vue
- BUT no code anywhere calls `showPrompt({ name: "compressStatusBar", pinned: true })`
- The design spec said "showPrompt when compression starts" but the implementation plan did NOT include a step to add this trigger
- This is a PLAN GAP - the plan had 10 tasks but none of them included the trigger code
- The CompressStatusBar component exists but is dead code - it will never be shown

**CompressImages.vue control buttons not visible:**
- Control buttons have `v-if="compressing"` condition
- `compressing` is set to true ONLY in:
  1. `confirmCompress()` -> starts compression (original code)
  2. `checkBackendStatus()` -> detects running backend (new code)
- BUT `checkBackendStatus()` is called AFTER the items guard:
  ```javascript
  mounted() {
    if (!this.items || !Array.isArray(this.items) || this.items.length === 0) {
      return;  // <-- RETURNS HERE, checkBackendStatus NEVER CALLED
    }
    this.backupEnabled = state.user?.compressBackup ?? true;
    this.expandItems();
    this.checkBackendStatus();  // <-- ONLY REACHED IF items EXIST
  },
  ```
- When reopening compress dialog (no items passed), mounted() returns early
- This is the SAME failure as v1.4.0.5's Bug B fix - the guard prevents status check
- The plan did NOT address moving checkBackendStatus BEFORE the items guard

**This is a critical implementation error.** The checkBackendStatus must be called regardless of whether items exist.

### 3.3 Fix Direction

1. **CompressStatusBar trigger:** Must add code to show CompressStatusBar when compression starts
   - Either in CompressImages.vue confirmCompress() (after starting compression)
   - Or in a global watch/poll mechanism
   - The showPrompt call must happen somewhere

2. **checkBackendStatus placement:** Must move BEFORE the items guard:
   ```javascript
   mounted() {
     this.checkBackendStatus();  // <-- ALWAYS check backend first
     if (!this.items || ...) {
       return;
     }
     // ... rest
   },
   ```

3. **Control buttons visibility:** Once checkBackendStatus runs and sets compressing=true, the v-if="compressing" buttons should appear

## Chapter 4: Compress Queue Interaction Broken (CRITICAL)

### 4.1 Symptom

When backend has an active compression queue:
1. User selects new files -> right-click -> Compress Images
2. Compress dialog opens briefly
3. Dialog disappears after a moment
4. Replaced by "compression status window" (the progress view)
5. Progress is shown BUT current file name is NOT displayed
6. No new status bar or control buttons appear

**MASTER Quote:** "如果距有后台压缩队列的情况下，选择文件进行压缩图片，距先喺出窗口，稍等瞬间压缩图片窗口消失，变成压缩状态窗口，就喺一般情况下压缩之后出现的状态窗口，然后距有进度，但喺当前文件名无显示的那种"
"变成了有队列就无法再进行压缩图片啦"

### 4.2 Analysis

**The dialog disappearing and becoming a status window:**
- This is likely caused by checkBackendStatus() being called (when items exist)
- checkBackendStatus detects running backend -> sets compressing=true -> shows progress UI
- But the progress UI replaces the file selection UI (because compressing=true hides the compress button and shows control buttons)
- Since no actual new compression was started, this is confusing

**Current file name not displayed:**
- checkBackendStatus sets `this.progressData = { total: status.total, current: status.processed, fileName: status.currentFile }`
- But if status.currentFile is empty or the polling hasn't updated yet, fileName will be empty
- Need to verify the API response includes currentFile

**Cannot start new compress when queue active:**
- Because compressing=true is set by checkBackendStatus, the compress button is hidden (v-if="!compressing")
- User has no way to add more files to the queue
- This is a UX design issue: checkBackendStatus should NOT prevent the user from selecting and compressing more files

### 4.3 Fix Direction

1. **checkBackendStatus should NOT prevent new compress:**
   - Show status info WITHOUT hiding the compress button
   - Or: Show status as a separate section/badge, not replacing the entire dialog
   - Or: Only show control buttons if user explicitly wants to manage the running job

2. **Dialog should not disappear:**
   - The compress file selection dialog should remain usable
   - Status info should be shown as additional info, not replace the dialog

3. **Current file name:** Ensure pollStatus API response includes currentFile field

### 7.2 Root Causes of Failure

**FAILURE 1: Plan did not include trigger code for CompressStatusBar**
- The design spec said "showPrompt({ name: 'compressStatusBar' }) when compression starts"
- But the implementation plan had NO step to add this trigger call
- Component was built, registered, but NEVER wired up
- This is a fundamental plan gap - building a component without wiring its trigger

**FAILURE 2: checkBackendStatus placed AFTER items guard (SAME mistake as v1.4.0.5)**
- mounted() has `if (!this.items || ...) return;` BEFORE checkBackendStatus()
- When reopening compress dialog without items, checkBackendStatus never runs
- This is the EXACT same bug as v1.4.0.5's reverted Bug B fix (commit 079cbeda)
- The plan even documented this as a risk but repeated the same mistake

**FAILURE 3: Bug H Fix 2 (setNavigationTransitioning skip) had unknown side effects**
- Adding `if (getters.previewType() !== 'image')` around setNavigationTransitioning in prev()/next()
- This was intended to skip the loading overlay for images
- BUT: isTransitioning state may affect button visibility CSS, transition timing, or other consumers
- No investigation was done on ALL consumers of isTransitioning before making this change
- Result: navigation buttons stopped displaying entirely

**FAILURE 4: No functional verification possible, but no trace analysis either**
- Docker-only deployment means no local testing
- All verification was grep + go build
- BUT: the plan should have included a "trigger trace" for each new component:
  - "What code path causes CompressStatusBar to display?"
  - "What code path causes checkBackendStatus to run?"
  - "What consumes isTransitioning and might be affected by skipping it?"
- None of these traces were done

**FAILURE 5: Tap navigation transition quality not investigated**
- The plan assumed removing @touchmove.prevent + conditional preventDefault would fix Bug G
- It did fix the tap functionality, but transition quality degraded
- Possible cause: the setNavigationTransitioning skip (Bug H Fix 2) affects ALL navigation including tap
- No investigation was done on why swipe is smooth but tap is not after the changes
